# /wrap — one-shot worker ship

Usage: `/wrap` (no args). Run from a worker session in a worktree when the work is ready for a PR.

Replaces the manual ship-time bash script. The skill captures start/finish timestamps, prompts for summary + test plan, writes `.pr-body.md` in the shape `/post-merge` parses, then pushes and opens (or updates) the PR.

**Why this exists.** Hand-rolled metrics blocks drift — workers forget the shape, paste placeholders, or skip the section entirely, which silently breaks `/post-merge`'s parsing. `/wrap` enforces the shape so `/post-merge` can parse cleanly every time.

**What `/wrap` does NOT capture: tokens + model.** Claude Code's `/cost` and `/model` commands render their output in a TUI popup, not in chat — there's no programmatic way for the skill to read those values. Leaving those Notion columns null is the honest answer for now. If you want tokens or model in the Notion task, backfill manually after merge.

## Preconditions

- Running in a worktree (cwd contains `simplesat-vision-worktrees/svp<N>-`).
- `.worker-meta` exists with the session start timestamp (written by the worker-bootstrap on the first user message).
- Branch has at least one commit beyond `main` and is checked out.
- `gh` authenticated.

If `.worker-meta` is missing, stop and tell Cory:
> `.worker-meta` not found — was this session started via `/spawn`? Capture the start time manually with `date -Iseconds > .worker-meta` and try again. (Note: it'll under-count the actual session duration.)

## Steps

### 1. Gather context

```bash
START=$(cat .worker-meta)
END=$(date -Iseconds)
BRANCH=$(git branch --show-current)
LAST_SUBJECT=$(git log -1 --pretty=%s)
```

Check for an existing PR for this branch:
```bash
gh pr list --head "$BRANCH" --json number,url,state --jq '.[0]'
```

If a PR already exists and is `OPEN`, we'll update it (`gh pr edit --body-file`) instead of creating a new one. If it's `MERGED` or `CLOSED`, stop and tell Cory — something's off.

### 2. Prompt for PR title

Show the worker the inferred title (last commit subject) and ask:

> **PR title** (press Enter to accept `<LAST_SUBJECT>`, or type a new one):

Keep it under ~70 chars. Don't auto-prefix anything — the commit subject convention already includes the type tag (e.g. `feat(skill): SVP-137 — /wrap skill`).

### 3. Prompt for summary

> **Summary** — 1–3 bullets of what shipped. One bullet per line. Empty line to finish:

Collect into a list. If the worker just hits Enter immediately, fall back to a one-line summary derived from the last commit body (`git log -1 --pretty=%b` first non-empty line), or from the brief title if even that's empty.

### 4. Prompt for test plan

> **Test plan** — what you tested. One bullet per line. Empty line to finish:

3–5 bullets is typical. Each becomes a `- [ ] <bullet>` checkbox in the PR body. If the worker types nothing, write a single `- [ ] Manual verification` placeholder so the section isn't empty.

### 5. Construct `.pr-body.md`

Exact shape — `/post-merge` parses this by awk-section + line prefix, so the format is load-bearing:

```markdown
## Summary

- <bullet 1>
- <bullet 2>

## Test plan

- [ ] <test 1>
- [ ] <test 2>

## Worker metrics

- Started: <START>
- Finished: <END>
- Tokens:
- Model:
```

Notes:
- Tokens and Model lines are intentionally blank — `/wrap` cannot read `/cost` or `/model` (TUI popups, not chat). `/post-merge` treats blank as null. Cory backfills manually in Notion if he wants those columns populated.
- Don't add a trailing `Generated by Claude Code` footer — Cory finds them noisy.
- Don't add emoji to section headers.

Write to `.pr-body.md` (gitignored) via the Write tool. Overwriting is fine — idempotent.

### 6. Push + open or update PR

```bash
git push -u origin "$BRANCH"
```

Then either:
- **No existing PR**: `gh pr create --title "<title>" --body-file .pr-body.md`
- **PR already open**: `gh pr edit <num> --body-file .pr-body.md` (and `gh pr edit <num> --title "<title>"` if title changed)

Capture the PR URL from the create/edit output.

### 7. End-of-turn output

One block, in the project's standard status-block shape (CLAUDE.md → Definition of done):

```
🟢 Shipped.

**Status:** PR #N open on `<branch>`
**Verified:** <whatever the worker said in test plan, condensed>
**Dev URL:** <if applicable, deep-linked localhost URL>
**PR URL:** <https://github.com/.../pull/N>
**Next step:** review and merge PR #N
**Risks/follow-ups:** none
```

Do NOT add a "tokens/model missing" warning — that's the expected state now. Cory knows.

## Idempotency

Safe to re-run. Common cases:
- **`.pr-body.md` already exists**: overwrite it. It's gitignored.
- **PR already open for this branch**: `gh pr edit --body-file` instead of `gh pr create`. The body refreshes; the PR stays.
- **PR is MERGED or CLOSED**: stop and surface to Cory. Re-shipping a closed PR almost certainly isn't intended.
- **Branch is behind origin**: `git push` will refuse non-fast-forward. Surface the conflict; don't `--force`.

## Escape hatch: `--body-from <file>`

If the worker has pre-written the PR body somewhere (e.g. drafted it during plan mode), they can skip steps 3–4 by invoking:

```
/wrap --body-from <path-to-markdown>
```

The skill still runs step 1 (timestamps), then **appends** the `## Worker metrics` section to the contents of `<path-to-markdown>`. The body file should already contain `## Summary` and `## Test plan` sections; the skill doesn't validate that, just appends.

## When NOT to use /wrap

- Supervisor-session PRs (doc edits to `main`, hot-fixes from `main`). The supervisor doesn't have a `.worker-meta` and the metrics block doesn't apply. Use plain `gh pr create` from `main`.
- Mid-work checkpoint pushes where you're not actually ready for review. Use plain `git push` to surface to `/sweep`, which will note "branch pushed, no PR yet" without opening one.
