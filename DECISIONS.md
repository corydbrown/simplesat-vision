# Decisions log

Assumptions and judgment calls made during the build. Items here were not explicitly specified by the original brief, but were chosen to keep momentum. Anything in this file is up for discussion.

## Stack and tooling

- **Next.js 16.2.6** (not 15). The brief said Next.js 15, but `create-next-app@latest` now installs 16 as the current stable. Next.js 16 brings two relevant breaking changes: `params` and `searchParams` are now Promises (must be awaited), and `next lint` is removed (we use ESLint's CLI directly via the `lint` script). Both are accommodated in the code.
- **React 19.2** (paired with Next 16).
- **Tailwind v4** with `@tailwindcss/postcss`, the create-next-app default.
- **shadcn/ui preset**: `radix-nova` (Lucide icons + Geist font), neutral base color, CSS variables, light mode default. Dark mode CSS variables are present so phase 2 can flip a class on `<html>` to support it.
- **Package manager**: npm. Only one installed locally; Vercel handles npm natively.
- **Node engines**: >=20 (current install is 25.8.1). Next.js 16 requires Node 20.9+.

## Data and IDs

- **Drizzle dialect**: `better-sqlite3` (synchronous driver). Drizzle config writes to `db/simplesat.db`. Migrations live in `drizzle/` and are checked in.
- **ID format**: `nanoid` (12-char URL-safe alphabet) with the prefixes from the brief: `cus_`, `tm_`, `tkt_`, `rsp_`, `qa_`.
- **JSON columns** (`tags`, `conversation`, `breakdown`): use Drizzle's `text({ mode: 'json' }).$type<...>()` for type-safe JSON.
- **Seed determinism**: `faker.seed(42)` so re-running `db:reset` produces the same data. Helpful for reviewer comparison.
- **Helpdesk distribution**: per direction from Cory mid-scaffold, **all seeded tickets are Zendesk**. The schema enum still includes gladly/gorgias/intercom for forward-compat, but no rows reference them.

## App structure

- **App Router** with `src/` and a `(workspace)` route group so the sidebar layout wraps all top-level routes (`/`, `/tickets`, `/responses`, etc.) without forcing the ticket detail route to also include it (it does, but cleanly).
- **State**: URL `searchParams` only for table sort/pagination. No state library, no client-side store.
- **Date/number formatting**: `Intl.DateTimeFormat` and `Intl.RelativeTimeFormat` in `src/lib/format.ts`. No `date-fns`, `dayjs`, etc.

## Scope cuts (phase 2)

- View tabs on the Tickets page render but only "All tickets" is active; the others are static.
- Toolbar (search, filter, group, sort, properties, export, new) is visual only in phase 1.
- Ticket detail is a full page route at `/tickets/[id]`, not a drawer. Drawer is phase 2.
- Customers / Team members / Responses / Reports pages are placeholder stubs.
- QA Evaluations: schema only; no seed data, no UI beyond the dashed "Coming Soon" card.
- No auth, no multi-tenant, no tests, no Storybook.

## Phase 3 decisions

- **Property registry**: each entity gets a single `properties.tsx` registry that powers both the list table AND the detail PropertiesPanel. Same source of truth; separate visibility persistence per `tableId`.
- **Properties visibility**: list view (`tableId="<entity>"`), detail panel (`tableId="<entity>-detail"`), and embedded tabs (`tableId="<entity>-tickets"`, etc.) each have separate column state in localStorage. User can show/hide independently per context (Notion-like).
- **Drawer architecture (phase 3)**: Next.js parallel intercepting routes (`@drawer/(.)[id]`). Same `<EntityDetailBody>` powers both the standalone `/<entity>/[id]` page (direct nav) and the drawer (intercepted from list). **Superseded in phase 4** — see below.
- **No sticky columns**: removed for now per Cory's feedback. To be reintroduced as user-configurable, not registry-hardcoded.
- **CompanyPill**: text only, no colored dot, no popover. Company is a string on Customer, not a first-class entity (yet).
- **Hover popover lifecycle**: lazy-mount only when `HoverCardContent` opens (Radix unmounts content on close by default). Module-level cache means each entity is fetched at most once per session. Critical for not firing 100+ fetches on every list page load.
- **Aggregate subqueries**: use literal `"<table>.<column>"` SQL inside `sql\`\`` correlated subqueries. `${schema.table.column}` interpolates as a parameter placeholder bound to nothing → NULL rows. (Bit me hard on listCustomers / listTeamMembers in phase 2.)
- **Multi-question survey schema**: `responses.answers` JSON column with `SurveyAnswer` discriminated union. `responses.rating` and `responses.comment` stay denormalized for fast indexed filtering used by Views and insights.

## Phase 3 scope cuts

- Reports / pivot table: planning doc only (`REPORTS.md`); execution deferred.
- Search wired to DB: search input is visual-only.
- Filter chip builder: visual-only (Views cover most cases).
- Saved-view creation: still curated in `src/lib/views.ts`, not user-editable.
- QA Evaluations full mockup: still a "Soon" card.

## Phase 4 decisions

- **Drawer architecture rewrite (route-based → URL-param-controlled)**. Phase 3's parallel intercepting routes only worked when a pill was clicked from inside its own section (`/customers/...` → customer drawer). Cory needed a drawer that opens from any page, swaps content in place without re-animating, supports back/forward through drawer history, and lets ESC + outside-click + X reliably close. The intercepting-route model couldn't do those things cleanly. Replaced with a single `<GlobalDrawer />` mounted in the workspace layout, reading `?drawer=<entity>:<id>` (and `?dt=<tab>` for in-drawer tabs) from URL state. Drawer body data is fetched client-side from new `/api/drawer/<entity>/[id]` routes. All `@drawer/` parallel-route folders deleted. See CLAUDE.md "Drawer architecture" for the current design.
- **Row click opens drawer, not standalone page**. EntityTable's `rowHrefBase` prop (which navigated to `/<entity>/<id>`) is gone. Replaced with `drawerEntity` — when set, row click pushes `?drawer=<entity>:<id>`. Used on list pages and on relation tables inside drawers. Pills inside rows keep their own click behavior; row handler skips clicks whose target is inside `a, button, [role='button']`.
- **`?dt=` and `?d*=` namespacing**. Drawer-internal tabs use `?dt=` so they don't collide with page-level `?tab=`. Drawer-internal table sort/page state is prefixed `d` (`dsort`, `ddir`, `dpage`) for the same reason. The default `dt` is materialized into the URL on drawer open (via `router.replace`) so refresh / share / back-forward all preserve drawer state including the selected tab.
- **Pill `DrawerLink` must forwardRef + spread props**. Phase 3 had a function-component `DrawerLink` that didn't forward refs or spread arbitrary props. Radix's `HoverCardTrigger asChild` silently dropped its injected pointer handlers, so popovers stopped working. Fixed by making `DrawerLink` a `forwardRef` component that spreads `...rest` onto the underlying `<Link>`. Documented in CLAUDE.md so future patterns don't reintroduce it.
- **Outside-click drawer dismissal ignores open Radix poppers**. When any Radix popper (dropdown, tooltip, hover card) is mounted in the DOM, the drawer's outside-click handler treats the click as dismissing the popper, not the drawer. Without this, clicking a dropdown's outside-click area also closed the drawer. Check is `document.querySelector("[data-radix-popper-content-wrapper]")` at the moment of mousedown.
- **Topbar back/forward**. Phase 3 had a single `<Link>`-based back button hardcoded per page. Phase 4 swaps to a `<HistoryNav>` with back + forward buttons that call `router.back()` / `router.forward()` — naturally cycles through drawer state changes and full-page navigations alike.
- **Response is now a first-class entity**. Has its own popover (`/api/popover/response/[id]`), drawer (`/api/drawer/response/[id]`), and `ResponsePill(id, rating, scale)` variant that's clickable. Plain `ResponsePill(rating, scale)` without `id` is still used in static contexts (ticket detail body) and renders as a non-interactive span.
- **Font color rule for table cell values**: `text-foreground` for primary identifiers (name, subject, company, comment text), `text-muted-foreground` for secondary metadata (emails, IDs, counts, dates, types), colored pill for stateful values (status, channel, tier, rating). Applied across all property registries; previous `text-foreground/80` opacity hacks removed.
- **No forced uppercase on labels**. Property names, group labels, section titles, column headers, answer-type chips: all natural sentence case now. `uppercase tracking-*` classes removed in phase 4. Decision was driven by readability + Notion-aesthetic feedback.
- **No vertical column borders in tables**. Removed `border-r` from cells and header cells; only horizontal row dividers remain.
- **Properties panel is borderless**. Phase 3 wrapped the panel in a rounded card with dividers between groups. Phase 4 drops the card and dividers; the panel reads as a flat list with light group separation via spacing only.
- **Drawer chrome layout**. Left side: close (`ChevronsRight`, also Esc) + maximize (`Maximize2`, opens standalone). Right side: copy link (clipboard, tooltip shows "Copied" for 1.2s) + kebab `DropdownMenu` with placeholder actions (Edit / Duplicate / Archive / Delete, all disabled — wired up later when actions exist). All four icons have tooltips with `Kbd` shortcut hints (new `src/components/ui/kbd.tsx` component, picks up shadcn's tooltip-inset styling automatically via `data-slot="kbd"`).
- **Detail header design**. Avatar + `text-3xl` name + small subtitle line (e.g. `company · email` for customers, `role · team · email` for team members). Internal id no longer rendered in detail-page headers — it stays available in the properties panel under "Identity". Ticket header keeps subject as h1 with `External ID: <num>` as a small muted line beneath.
- **Pill alignment trick**. Interactive pills (Customer, TeamMember, Ticket, Response-with-id) use `-mx-1 px-1`. The `px-1` gives the hover background some breathing room around the text; the `-mx-1` pulls the text content back to align exactly with the column header. CompanyPill (text-only, no hover state) has neither.

## Phase 5 decisions (Reports pivot editor)

- **No pivot library**. Evaluated react-pivottable, WebDataRocks, AG Grid (enterprise), and @tanstack/react-table. All either clash with the shadcn/Notion aesthetic, are paid, or don't actually do cross-tabbing. Built custom: SQL `GROUP BY rows ∪ columns` via Drizzle, ~70 lines of cross-tab in `src/lib/reports/pivot.ts`, render with shadcn `Table`. Matches CLAUDE.md "Component reuse > custom CSS". Total bundle impact: zero new deps.
- **URL state, not localStorage**. `?r=<base64(JSON)>` holds the full `ReportConfig`. Shareable; back/forward works; refresh preserves state. localStorage holds *only* the rail width. Saved-report persistence is deferred to a future phase but slots in cleanly because URL state is already the source of truth.
- **Two metadata layers**. The visual `Property<T>` registries in `src/lib/properties/<entity>.tsx` are "use client" and own table/detail rendering. Reports has its own server-friendly registry at `src/lib/reports/pivot-fields.ts` keyed on the same `propertyId`. Reason: compile.ts is server-only and the visual registries reference client components. Pivot-fields holds the SQL info (`groupExpr`, `labelExpr`, `joins`, `valueOnly`, `aggregations`) — visual registries hold the JSX. No coupling required between them at runtime; both keyed on `propertyId`.
- **Constraints: 2 rows · 1 column · 3 values**. Single column axis was a deliberate UX call — multi-column cross-tabs get unreadable fast. Two rows lets you nest (e.g., tier → team member). Three values is plenty for any one view.
- **`valueOnly` fields**. Customers and team_members bases need computed metrics (`total_tickets`, `avg_rating`) via correlated subqueries. Those can't be used as group keys (`GROUP BY` over a per-row subquery is meaningless), so they're flagged `valueOnly: true` and the UI hides them from Rows/Columns. They're allowed in Values and Filters where the subquery resolves naturally.
- **Agg-only computed fields don't appear in Rows/Columns**. UI filters them out in the rail's default-axis routing, the AddFieldButton popover (`allowValueOnly={false}` default; `true` only on the Values zone), and `compile.ts → addAxis` guards as belt-and-suspenders.
- **No `*` propertyId for agg fields**. Count-of-records uses the sentinel `propertyId: "*"` with `agg: "count"`. Any other agg requires a real propertyId. This keeps the type narrow.
- **"Team member" rename, project-wide**. Renamed everywhere user-facing (`Agent → Team member`). Renamed in property IDs too (`agent → team_member`) — that's part of the encoded URL state, so the rename invalidates old shared URLs. Acceptable at prototype stage. Kept the conversation-message `role: "agent"` enum in the schema (different concept: who sent the message in a thread).
- **"Assignee" vs "Team member" labels**. On the Tickets base, the joined team_members fields are labeled "Assignee" because that's the Simplesat-side framing of the ticket-attribute. On the Responses base, they're labeled "Team member" because that's the canonical entity. Both point to the same `team_members` table.
- **Entity-based property grouping**. Rail and Add Field popovers group by source entity, not by registry concept. `GROUP_ORDER` per base ensures consistent ordering (primary entity first, joined entities after). Replaced the prior generic groups ("Identity / State / Source / Survey / Activity / Relations") which mixed entity concerns.
- **Inline axis row**. Single horizontal strip for Rows / Columns / Values / Filters with vertical dividers, wraps on narrow screens. ~40% the vertical footprint of the previous 4-row layout. Each axis is its own `useDroppable` target.
- **Base entity dropdown in toolbar, not rail**. Most reports pivot Responses; the picker doesn't deserve permanent vertical real estate. Moved to a `BaseEntityDropdown` ("Pivot over Responses ▾") in the toolbar. Default is Responses. Rail is now Properties-only.
- **AlertDialog confirms base switch**. Replaced `window.confirm()` (which clashes with the design system) with a shadcn AlertDialog. Only triggers when there's existing config to wipe.
- **Allow zero values**. Previously the builder always kept at least one value (defaulting to count). Now values can be empty; the pivot table is replaced with the empty state. A separate "Count" quick-add button surfaces in the Values area when `*` isn't already there, making the common case one click.
- **DragOverlay for drag visual**. dnd-kit's source element only fades; the chip preview that follows the cursor is rendered via `<DragOverlay>` portal with `dropAnimation: null`. Necessary for legibility — without it, users couldn't see what they were dragging.
- **Rowspan grouping for 2-row pivots**. When two row axes share a first-axis value across consecutive rows, the first `<th>` uses `rowSpan` and subsequent rows skip the cell. Notion / Excel behavior — each group reads as a labeled block. Computed via a pure `computeRowSpans()` pass on row labels.
- **Pivot table is `w-auto min-w-full`, not `w-full`**. Lets the table grow beyond container width when columns genuinely need more room (horizontal scroll kicks in) but otherwise fits exactly. `overflow-x-auto overflow-y-auto` on the wrapper.
- **AI prompt-to-config is stubbed**. UI is fully wired (`AiPromptDialog` → server action `buildReportFromPrompt`). Today the action returns a hardcoded ReportConfig based on simple keyword matches against a handful of test prompts. Real Anthropic call is queued for the next session — see `REPORTS.md → AI build-with-prompt`.

## Phase 6 decisions (AI prompt-to-config + multi-metric reports)

- **Anthropic SDK, real call.** Replaced the keyword-stub `buildReportFromPrompt` with `@anthropic-ai/sdk` + `claude-haiku-4-5` + forced tool use. The tool's `input_schema` mirrors `ReportConfig` (enums for `agg` / `bucket` / `op`, `maxItems` for axes), so structured output is guaranteed — no JSON parsing. `tool_choice: { type: "tool", name: "build_report" }` makes the call always emit the tool. Model choice: Haiku is plenty for ~25 fields and a tight schema; bump to Sonnet 4.6 only if accuracy regresses.
- **Per-base system prompts, cached.** `buildSystemPrompt(base)` serializes the `PIVOT_FIELDS[base]` registry into the system block (id, label, dataType, aggregations, bucketable, enumValues, valueOnly). Cached with `cache_control: { type: "ephemeral" }`. Stable across calls per base ⇒ cache hits accrue naturally. May silently no-op if the prompt drops under the model's ~4096-token caching minimum; acceptable.
- **Sanitization, not trust.** `sanitize(base, candidate)` walks every output from the model and drops anything inconsistent with the registry — unknown propertyIds, `valueOnly` fields in rows/columns, aggs outside `field.aggregations`, ops outside `field.filterOps`. Clamps axis lengths. Falls back to `defaultConfig(base)` if everything got stripped. The model is treated as a noisy suggestion source, not a source of truth.
- **Inline error in dialog, no toast lib.** Errors from the server action surface as `text-destructive` below the input, prompt is preserved for retry. Sonner / a global toast system would be nice for the whole app but is out of scope here; one tiny inline message beats a new dependency for a single failure case.
- **Compile-time filter guard.** `compile.ts` independently drops any filter whose op isn't in `field.filterOps`. Belt-and-suspenders against stale `?r=` URLs, hand-edited state, and AI mistakes — the SQL boundary is structurally unreachable rather than prompt-policed. This is the load-bearing fix that caught the "AVG misuse" error from a saved URL with `csat_avg` in a WHERE on the response base.
- **Multi-metric reporting via metric-typed values, not a global metric selector.** Old debate: production Simplesat forces users to pick CSAT vs CES vs NPS at the report level because mixing them is mathematically nonsense. Considered three alternatives:
  - *Survey-type as an axis* — works for CSAT/CES/5-Star (all averages on the 1–5 scale) but fundamentally breaks NPS (a percentage, not an average; scale 0–10).
  - *Auto-segment one value into N* — mixed units in adjacent cells confuse readers; behavior is magic and hard to predict.
  - *Force-pick-one (today's prod)* — clean math, constrains the user.

  Picked **metric-typed values**: each metric is a first-class entry in the rail (`csat_avg`, `ces_avg`, `ces_positive_pct`, `nps_score`), each carries its own SQL formula. The user can put CSAT (avg) and NPS score side-by-side as separate columns and each is internally coherent. Cross-metric comparison just works.
- **CES "% positive" = top-2-box (rating ≥ 4).** Standard SaaS convention; matches Cory's call.
- **5-Star skipped this round.** The `survey_type` enum is currently `csat | nps | ces` — adding 5-Star is a one-line schema migration + seed update + one metric helper entry. Punted to keep this change focused.
- **Metric fields are `valueOnly`; filterability differs by base.** On the response base the formula is a direct aggregate (e.g. `AVG(CASE WHEN survey_type='csat' THEN rating END)`) and can't appear in WHERE → `filterOps: []`. On the customer / team_member / ticket bases the formula is a correlated subquery → row-level filterable → `filterOps: NUMERIC_OPS`. AI prompt teaches the model which base to use for which kind of filter.
- **Compiler uses `field.valueExpr` verbatim when present.** Pre-aggregated metric formulas bypass the `aggExpr` switch — the user's chosen agg is ignored because the metric encodes its own math. One-line change to `compile.ts`.
- **Seed now distributes survey types 65 / 20 / 15 (CSAT / CES / NPS).** Same `baseRating` (1–5) drives all three, remapped to 0–10 for NPS so the detractor/passive/promoter mix stays coherent. Without this, every new metric value would render as zero and the demo wouldn't show off the feature.
- **No prompt-only fix for "low CSAT" — split the work.** Cory's two failing prompts ("Companies with low CSAT score by quarter", "Top performing team members") could have been patched with prompt heuristics alone, but the underlying ambiguity (one `rating` field across multiple metric types) would keep biting. Did both: metric-typed values for structural correctness, plus prompt heuristics for sentiment / axis discipline / top-N behavior.
- **Sort-by-value is still a gap.** "Top performers" produces a clean, correct, *unsorted* table. Not blocking. Tracked under REPORTS.md "Not built yet" — column sort needs `compile.ts` (ORDER BY by value alias), `pivot-table.tsx` (sortable column header), and URL state for the sort direction.

## Phase 7 decisions (reseed from production shape)

Goal: replace the toy seed with one that demonstrates the property-explosion UX (35k customers, hundreds of synced fields), multiple survey types and channels, real topic taxonomy with sentiments, and real-feeling comments — without compromising on PII safety or seed runtime.

- **First-class `surveys` entity, no nav entry.** Added `surveys` table with metric/channel/scale/status/questions JSON. Pill + popover + drawer + standalone `/surveys/[id]` page wired the full new-entity walkthrough. **Intentionally not in PrimaryNav** — per Cory, survey management lives in settings later. The standalone route exists because pills need a cmd-click target; the absence of a `/surveys` list route is the signal that this is a referenced entity, not a navigated one.
- **`responses.surveyId` FK + denormalized `responses.surveyType`.** The denormalization is load-bearing: the metric formulas in `pivot-fields.ts` (`csat_avg`, `nps_score`, etc.) already filter by `survey_type` directly on the responses table for fast indexed aggregation. Moving the metric to `surveys.metric`-only would force every metric pivot to JOIN surveys, breaking existing report URLs and slowing down the response base. Both fields exist; seed keeps them in sync at write time.
- **`SurveyType` enum expanded** to `csat | nps | ces | five_star | custom`. Schema migration changes the enum constraint on `responses.survey_type` and adds the new survey entity. Older seed buckets (`csat | nps | ces` only) no longer represent prod-realistic distributions.
- **5-Star metric is now a first-class pivot value.** `five_star_avg = AVG(rating WHERE survey_type='five_star')`, mirrors the CSAT formula. Filterability differs by base (direct aggregate on responses → `filterOps: []`; correlated subquery on customer/team_member/ticket → `NUMERIC_OPS`) per the existing pattern.
- **Custom properties as JSON, not a `property_definitions` table.** Considered three shapes:
  - *EAV (`customer_properties (customer_id, key, value)` table)* — clean, queryable, joins are correct. But Reports compiler would need a complex EAV-pivot for each field, and the rail would have to be DB-driven. Killed.
  - *Column-per-field on the customers table* — fastest queries, simplest pivot. But the table balloons to ~70 columns of sparse data and migrations get awful.
  - *JSON column + TS const registry* (picked) — `customers.customProperties` JSON, definitions in `src/lib/properties/custom-fields.ts`. Sparse by design (each customer carries 30-60 of ~65 keys). Reports pivot fields use `json_extract(custom_properties, '$.id')`. Importance drives display ordering. Hand-tunable for the demo narrative.
- **Importance ≥3 surfaces in Reports rail; everything still in column picker.** 65 customer custom fields would flood the pivot rail with low-signal items (Twitter followers, card brand, ConnectWise location). Filtering to importance ≥3 keeps Reports usable. The column picker on EntityTable shows everything because hide-by-default + grouped-by-source makes long lists tolerable there.
- **Region / language / Zendesk group promoted to top-level team_member columns** (not in customProperties). They're filterable on every entity table including ticket joins, used in pivots as enums, and shouldn't live behind a JSON extraction. The Zendesk integration would canonically populate these, so they sit at the row level. Other Zendesk fields stay in customProperties.
- **Topic taxonomy as a TS const, not a DB table.** 68 topics × 20 groups is small, stable, and curated. A `topics` table would force every topic lookup through a JOIN and complicate the seed without adding flexibility we need. The taxonomy comes from `csv_exports/topics_groups.csv` and is regenerated by hand into `src/lib/topics.ts` when production changes.
- **Per-answer + roll-up topics.** Each `SurveyAnswer` carries optional `topics: {topic, sentiment}[]`. Response-level `topics` is the deduped roll-up across all answers. Conflict resolution: negative > neutral > positive (worst signal wins). This mirrors what Cory described — the API exposes per-answer topics, the UI rolls them up for display, and conflicting sentiment on the same topic should bias toward the negative read.
- **Topic-as-pivot-axis deferred.** Topics live in a JSON array; using them as row/column axes needs a `CROSS JOIN json_each(topics)` pattern the existing compiler doesn't support (it would change row cardinality from "one row per response" to "one row per (response, topic) pair"). Tracked as a follow-up. For now topics are storable + visible + filterable via UI components but not pivotable.
- **Comment harvest with two-layer PII scrub.** `scripts/harvest-comments.ts` (a) drops comments containing any harvested full-name span (≥2 capitalized tokens) or company name from the export, then (b) substitutes single-name tokens with `[name]` via word-bounded regex. The single-token replace prevents "Angelika was very helpful" from leaking through the full-name filter. Comments with >25% `[name]` tokens after substitution get dropped (over-scrubbed reads as obviously synthetic). 533 unique comments survived from 3,221 rows, bucketed by metric × rating. Spot-checked by hand.
- **CES bucket fallback.** Production export had no CES feedback in the sampled period, so the harvest script remaps CSAT 1-5 comments into CES 1-5 buckets. Acceptable because CSAT and CES use the same 5-scale and the comment language is interchangeable ("Was easy", "Took too long" reads as both satisfaction and effort).
- **Seed scale: 1,200 customers / 50k tickets / 14k responses.** Considered going to 5k or 10k customers to demo "real production scale," but the UI patterns (paging, search, filtering, property-importance ordering) are indistinguishable past ~1k. Bigger seed = slower `db:reset` for no perceptual gain. 50k tickets stays from the prior seed because that's where insights queries get visibly slow if anything goes wrong — useful signal.
- **8 surveys, weighted to mirror production.** Intercom CSAT and Zendesk CSAT carry the bulk (~40% combined), NPS quarterly + web-embed share 20%, CES gets 9%, 5-Star onboarding 8%, custom roadmap 5%. The weighting is a per-response pool, so customers can have responses across many surveys.
- **Survey questions are stored as JSON, not a separate `survey_questions` table.** Surveys are read-only in this prototype and questions are tightly coupled to their parent survey. JSON keeps the question definitions next to the survey and avoids a normalization that buys nothing.
- **Response answers seeded from real survey definitions, not a generic 3-question template.** The previous seed hardcoded "rating + multi-choice + multi-select + comment" for every response. Now `buildSurveyAnswers` walks the actual survey's questions and emits matching answers, biased by the (clamped 1-5) sentiment of the rating. The roadmap survey's multi-select demo of "which areas to invest in" produces realistic-looking aggregations in Reports.

## Phase 7 known gotchas

- **Partial npm packages.** Halfway through phase 7, `class-variance-authority` and similar packages had their `package.json` installed but the `dist/` directory missing — pure typecheck failed across `ui/button.tsx`, `ui/badge.tsx`, etc. Fix: `rm -rf node_modules && npm install`. Possibly triggered by interleaving `npm install --save-dev csv-parse` with concurrent file edits.
- **`ALTER TABLE ADD COLUMN NOT NULL` without a default** requires the table to be empty when applied. Migration `0002_empty_randall.sql` adds `responses.survey_id` as `NOT NULL` — works because `db:reset` blasts the DB file before running migrations, but a future incremental migration on an existing populated DB would need a backfill step.

## Phase 4 known gotchas

- `next dev` (Turbopack) persistent cache (`.next/dev/cache/turbopack/`) can corrupt when files are deleted while the dev server is running (we deleted four `@drawer/` route folders mid-session and triggered this twice). Symptoms: `TurbopackInternalError: Failed to restore task data`, or `Cannot find module '.next/dev/...'`. Fix: kill `next dev`, `rm -rf .next`, restart. Documented for Cory in conversation; no permanent code change needed.
- `eslint-config-next@16.x` occasionally ships without `dist/` contents. If lint fails with `Cannot find module 'eslint-config-next/dist/core-web-vitals.js'`, nuke `node_modules` + `package-lock.json` and `npm install` again. Hit once in phase 4.

## Phase 8 decisions (B2C beauty retail reseed)

Goal: pivot the prototype's narrative from B2B SaaS to mid-market B2C beauty retail (Sephora-style — "Bloom Beauty"), correct the way custom attributes are modelled to match Simplesat's real API, and add team-member groups + ticket priority to match the Zendesk-shaped data model the production product mirrors.

- **Brand**: "Bloom Beauty" everywhere user-visible. Picked over Sephora/Abercrombie alternatives because (a) loyalty tier maps cleanly onto the existing three-bucket tier enum and (b) the property-explosion UX demos better with beauty-personalization fields (skin type, shade, allergies, app installed) than apparel sizing.
- **Loyalty tier rotation**: `starter | pro | enterprise` → `insider | gold | elite`. Beauty-loyalty-flavored without being a literal clone of Sephora's tier names. Migration via table rebuild (SQLite CHECK constraint rebuild). Seed reset is the migration path — no production data, no backfill required. Old `?r=` pivot URLs that referenced the old enum values are invalidated; acceptable at prototype stage.
- **Mid-market B2C, ~95% individuals / ~5% B2B accounts.** `customers.company` is now nullable. ## Phase 8 known gotchas

- **Drizzle `INSERT INTO __new_customers ... SELECT ... FROM customers` for nullable-column conversion.** Drizzle-kit's generated migration referenced the new `language` / `company_external_id` / `company_domain` columns in the SELECT, but those columns don't exist on the old table — the INSERT would fail on the SQLite parser, even against an empty table. Patched by hand to use `NULL AS "language"` etc. in the SELECT. Same pattern likely needed any time a column-rebuild migration adds new NOT-NULL or default-bearing columns alongside a constraint change.
- **Drizzle-kit's interactive prompt for column rename detection** doesn't surrender when stdin isn't a TTY. Driven via `expect` to auto-confirm "create column" over "rename column" for `zendesk_group → group_id`. Documented for future migrations that drop+add semantically distinct columns on the same table.
- **`@import "shadcn/tailwind.css"` doesn't resolve under Next.js 16 + Turbopack PostCSS pipeline.** The `shadcn` npm package exposes `./tailwind.css` only under the `style` export condition, which Turbopack's PostCSS resolver doesn't honor. Worked around by copying `node_modules/shadcn/dist/tailwind.css` into `src/app/_shadcn-tailwind.css` (95 lines, just keyframes + custom variants) and importing locally. If you upgrade the shadcn package and want fresh content, re-copy from the dist file — don't try to "fix" the import back to the canonical subpath form unless Turbopack starts honoring `style` conditions.

## Phase 9 decisions (deploy to Vercel)

Repo went public on GitHub; app deployed to https://simplesat-vision.vercel.app.

- **DB host: Turso (libSQL) via Vercel marketplace integration.** Picked over Neon/Vercel Postgres because the codebase is already SQLite-shaped (Drizzle + SQLite SQL functions like `json_extract`, correlated subqueries that assume SQLite semantics). Migrating dialects would have meant rewriting the metric formulas in `pivot-fields.ts` and the per-view predicates. libSQL is the same SQL surface, just hosted. Free tier (Starter plan) is way more than this prototype needs.
- **`better-sqlite3` → `@libsql/client` swap.** `src/db/client.ts` rewritten to `createClient({ url, authToken })` + `drizzle-orm/libsql`. Reads `TURSO_DATABASE_URL` / `TURSO_AUTH_TOKEN`; falls back to `file:db/simplesat.db` so `npm run db:reset` keeps working unchanged for local dev. `drizzle.config.ts` switched to dialect `"turso"` with the same env fallback.
- **Async seed.** libSQL's driver is async; `better-sqlite3` was sync. `src/db/seed.ts` rewritten to await every write and convert the two bulk-insert transactions to `await db.transaction(async (tx) => ...)`. Took ~30s end-to-end against the file URL, ~60s against remote Turso for the same 1,200 customers / 50k tickets / 13.6k responses payload.
- **Seeding is a manual one-shot, not part of the build.** Considered wiring seed into `postbuild` so Vercel auto-seeds. Killed: it'd re-run on every deploy (60s for nothing), and the seed is destructive (`db.delete(...)` first). The intended flow is: provision Turso → set env vars in Vercel → run `set -a && source .env.production.local && set +a && npm run db:migrate && npm run db:seed` once from a laptop with the production credentials. Document the flow rather than automate it.
- **`vercel env pull` returns empty strings for env vars marked "Sensitive".** The Turso marketplace integration sets the injected vars as sensitive by default — same with anything added through the dashboard with the "Sensitive" toggle on. They work fine at runtime, but `vercel env pull` silently returns `""` for them. To get the live values you have to grab them from the Storage tab (or the provider's own dashboard / CLI). Documented because it cost us 10 minutes mid-seed.
- **PR-based workflow held for the migration.** Even though `main` is the "supervisor" branch per CLAUDE.md, the libSQL swap landed via a feature branch + PR (#1, "Swap better-sqlite3 → @libsql/client for Vercel deploy") rather than direct push to `main`. Cleaner diff for review, and consistent with the worktree-style workflow for any meaningful code change.

## Phase 10 decisions (design token system + Lato)

- **Design tokens moved to a dedicated `DESIGN.md`.** `CLAUDE.md` previously held the full token dictionary inline under "Visual tokens". Split into two: `DESIGN.md` owns the table-of-values (colors, typography wiring, border/state tiers, Figma mapping, "how to change a token"), and `CLAUDE.md` keeps only the *philosophy* (when to use `--primary` vs `--brand` vs `--positive`, why status colors are softer than Figma). Mirrors the existing doc pattern (`DECISIONS.md`, `REPORTS.md`, `AGENTS.md`) where each file has a single concern. Reason this matters: token edits are now a one-file diff, and designers can review `DESIGN.md` without wading through agent-guidance prose.
- **Lato via `next/font/google`, reversing the prior "no web font" rule.** `CLAUDE.md` previously said "Keep system font stack. Do not introduce Lato or any web font." Reversed at Cory's request to see how the prototype reads in the live app's typeface. Wired as `--font-lato` (Next.js font variable) → `--font-sans` (CSS chain in `globals.css`, with system fallback after) → `font-sans` (Tailwind utility). Zero component changes — every existing `font-sans` (and the implicit body inheritance) just resolves to Lato. Easy to revert: change `Lato` to system fallback in one place.
- **Architecture: three-layer token chain, components consume utilities only.** Codified the existing pattern explicitly in `DESIGN.md`: (1) raw value in `:root`, (2) `@theme inline` alias exposing it as a Tailwind utility, (3) component uses `bg-foo`/`text-foo`. Means: a future "change all borders darker" / "swap font to Inter" / "shift positive green warmer" is a one-line edit in `globals.css`. Components are the last layer to know about a token change.
- **New tokens added (additive, no breaking changes).** From the Simplesat Figma design system, mirrored into our token set:
  - Text tier: `--foreground` now `#373f46` (the design system's "Black", not pure black), `--foreground-light` (45%), `--foreground-disabled` (20%). `--muted-foreground` retuned to `#373f46a6` so all text tiers come from the same hue.
  - Brand states: `--primary-hover` (#0066e6), `--primary-down` (#0066cc) for explicit interaction states where `/N` opacity isn't sufficient.
  - Border tiers: `--border-strong` (20% black), `--border-solid` (#d4d4d5) for use over non-white surfaces.
  - Selection: `--selection` (#cce2ff), `--selection-foreground` for row-selected / text-highlight backgrounds (matches Simplesat's `highlightBackground`).
  - Canvas: `--canvas` (#f4f4f4). **Defined but not applied** — switching the body to `bg-canvas` would require migrating the two existing `bg-background` panels (`workspace/page.tsx` dashboard section, `topbar.tsx`) to `bg-card`. Deferred behind a "Pending migrations" note in `DESIGN.md`.

For a CSAT prototype, primary + status + Tailwind opacity utilities cover the surface. Adopting all of Figma's tokens would clutter the system and tempt off-pattern hue choices in components.

## Design system codification (post-audit, 2026-05-21)

The `/design` audit page (PR #12) surfaced three categories of drift that required architectural choices, not just code fixes. Decisions captured here; doc changes land in this PR (additive); code migration follows in a Kind-B mechanical sweep.

- **Stateful pills wire through the production hue palette, not state-semantic tokens.** Earlier the system carried `--positive` / `--negative` / `--neutral` / `--info` as aliases for mood-bearing values. The codebase never adopted them — every pill (status, priority, channel, tier, team, team-group, survey status) hardcoded raw Tailwind hues like `bg-red-50`. The deeper problem: state-semantic naming conflated "what state this is" with "what color this is" — labeling green as "positive" forced green into a single role even when it was just being decorative on a team-group pill. Decision: drop state-semantic aliases; map every pill directly to a production hue (`bg-green-lighter` for solved, `bg-red-lighter` for urgent, `bg-purple-lighter` for the VIP team-group). 7 chromatic hues × 5 shades, plus `--black` / `--white` / `--vanilla`. Tier 1 (raw hues) is the source of truth; Tier 2 (structural-semantic tokens like `--foreground`, `--background`, `--border`) flips per mode and aliases hues. State-semantic tokens stay deprecated and get removed during the mechanical sweep.
- **Type ladder shifted to 15 / 14 / 12** (text-base / text-sm / text-xs). Body content moves from 16 → 15. Pills move from `text-xs` (12) to `text-sm` (14) — the 12 → 14 jump matters because production-grade tools (Notion, Linear, GitHub) almost never go below 14 for body labels; the prototype was reading dense by default. `text-xs` (12) survives but for `kbd` and rare tight chrome only. The accompanying rule: **de-emphasis is via muted color, not smaller text.** Earlier guidance ("the answer is usually muted color, not smaller size") was strengthened from a hint to a rule. All `text-[Npx]` arbitrary one-offs (60+ in the codebase) get mapped to ladder steps or get the ladder extended; no arbitrary inline sizes.
- **Chart palette aliased to production hues.** The `--chart-1..5` greyscale `oklch` values were shadcn CLI installation defaults — unused, unintentional. Reports hasn't shipped yet, so nothing visible drifted. Repoint `--chart-1..5` (and new `--chart-6`) at `--blue` / `--green` / `--yellow` / `--red` / `--purple` / `--teal`. When Reports lands, colors are already correct.
- **Vanilla 5-shade interpolation.** Production palette shipped `--vanilla-darker` (`#666100`) and `--vanilla-lighter` (`#FEFCD5`) only — the middle three shades were missing. Filled in (`--vanilla-dark: #998C00`, `--vanilla: #CCBB33`, `--vanilla-light: #E6D87A`) for symmetry with the other hues. Values are interpolated guesses; tune in review.
- **Discipline rules tightened.** The audit caught patterns where the system was bypassed rather than the system being wrong (hand-rolled `<span className="text-[10px] uppercase tracking-wider">DEMO</span>` instead of `<Badge>`; pill components reinventing the shape inline; raw Tailwind hues like `bg-red-50` instead of `bg-red-lighter`). Codified as four rules in CLAUDE.md → Conventions: no reinventing components inline, no raw Tailwind hue classes, no arbitrary text sizes, muted color over smaller text. Same rules echoed in DESIGN.md → "Discipline rules" so token docs and agent docs stay in sync.
- **Did NOT adopt** opacity-based hover (`bg-X/N`) for hue-backed components. Production convention uses the next-darker step (`bg-blue-dark` for hover, `bg-blue-darker` for pressed). Avoids the text-contrast degradation that opacity-based hover introduces on light tints. Buttons, pills, and links should adopt this pattern during the mechanical sweep.
- **Did NOT adopt** Tailwind v4-replacement frameworks (CSS-in-JS, styled-components, etc.). The current Tailwind utilities + `@theme inline` + CSS vars chain IS the "components encapsulate styles + change via variables" pattern — moving off Tailwind would be a regression for marginal gain. The discipline rules above are the real fix; the foundation is sound.

## Phase 11 decisions (Notion-style group-by)

- **`groupable` is a per-property flag, not a derived enum-detection rule.** `Property<T>` gains `groupable`, `groupValue: (row) => string | null`, `groupLabel?`, and `nullGroupLabel?`. The registry decides per-column whether grouping is meaningful — same pattern as `sortable` and `filter`. Deliberate: free-text columns (subject, name, email) and unique IDs are never opted in, low-cardinality enums and relation labels are. Server-only field maps mirror the filter/sort convention under `src/lib/group/fields/<entity>.ts`.
- **URL shape: `?group=<propertyId>:<dir>`.** Comma-tuple style mirrors the multi-sort PR (#15) shape. Bare `?group=status` parses as `asc`. Drawer-mode tables use the existing `?dgroup=...` prefix. URL state is the source of truth — no localStorage persistence (grouping is a query state, not a preference).
- **Asc/desc only; manual + custom enum ordering punted to v2.** v1 sorts sections by the group column's natural order in the chosen direction. The right fix for status (where the meaningful order is `open → pending → solved → closed`, not alphabetical) is a `groupValueOrder?: string[]` on the property — registry-defined, not URL-defined. Not in this PR.
- **Composition with multi-sort (PR #15).** Server queries spread group orderBy expressions FIRST, then sort orderBy: `.orderBy(...compileGroupOrderBy(groupBy, FIELDS), ...buildOrderBy(sorts))`. Group prepends so rows in the same group stay contiguous; multi-sort applies within each group. Mirror on the client: `applyMultiSort(rows)` happens before bucketing by `groupValue`. Sort and group are orthogonal — sort works on its own, group works on its own, and combined the rows-within-section ordering honors the sort URL.
- **Sections render client-side; server pre-orders for paginated correctness.** EntityTable buckets rows by `groupValue(row)` regardless of server order. The server's `compileGroupOrderBy` forces nulls last via `CASE WHEN ... IS NULL THEN 1 ELSE 0 END`. Embedded detail tables don't plumb groupBy through the drawer API — they rely on the client-side bucketing since they're not paginated.
- **Counts in section headers are "this page" counts, not group totals.** Honest single source of truth. Only /tickets paginates, and the friction of a separate per-group aggregate query wasn't worth the marginal accuracy. The user can filter to a specific group if they need an exact total.
- **No × on the active-group chip.** Removal happens via the popover's "Remove grouping" action. The chip is pure display — keeps the click target single (open the popover) instead of one-affordance-too-many.
- **Group control placement.** Toolbar: `Filter → Sort → Group → Properties`. Embedded detail tables: SortControl + GroupControl sit next to the existing `OpenInTable` in the `RelationTabs` trailing slot (customer detail, team-member detail) or in the section header trailing slot (survey detail). No standalone toolbar above embedded tables — there isn't one to put it in.

### Phase 11 deferred (v2)

- **Date-as-bucket grouping** (Today / This week / This month / Older). Architected for via the `groupValue: (row) => string | null` accessor — just hasn't been wired for any date column yet. Needs a `bucketDate` helper + decisions on bucket boundaries per column.
- **Custom-field grouping.** Custom fields don't yet declare `groupable` on a per-definition basis. Need to extend `CustomFieldDef` with eligibility metadata (which kinds of custom fields make sense as group keys) and wire `customFieldProperties()` to thread it through.
- **Custom enum ordering** (status: open → pending → solved → closed instead of alphabetical). Plan: add `groupValueOrder?: string[]` to `Property<T>`, override the section-sort and emit a `CASE WHEN col = 'open' THEN 0 WHEN col = 'pending' THEN 1 ... END` order expression server-side.
- **Multi-property / nested grouping.** "Group by Status, then by Priority." URL shape would extend to comma-list. Section render becomes a tree.
- **Collapsible sections.** Notion-style click-to-collapse with row-count visible when collapsed. Section state in localStorage per `tableId`.
- **Group totals in section headers.** Replace "rows on this page" with an aggregate-query total via a separate count-by-group query. Mostly meaningful on /tickets.
